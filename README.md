# dizhi123a

自动节点更新与替换系统

## 功能概述

这是一个 GitHub Actions 工作流，用于自动从指定源获取网络节点信息，并进行智能化的 IP/端口替换，最终生成合并的节点文件。

## 主要特性

- 🕐 每8小时自动运行一次，也支持手动触发
- 🌐 从多个源获取不同类型的节点（VMess、VLESS、SS/SSR）
- 🔄 智能IP/端口替换机制
- 📊 节点新鲜度追踪与报告
- 🚫 自动过滤包含特定标记的节点

## 工作流程

### 1. 节点获取与分类

- 从 `VMESS_URL` 和 `SS_URL` 获取原始节点数据
- 清理HTML标签并提取节点链接
- 按协议分类存储：
  - `dizhi1.txt`：VMess 节点
  - `dizhi2.txt`：VLESS 节点
  - `dizhi3.txt`：SS/SSR 节点

### 2. IP/端口替换逻辑

这是系统的核心功能，详细说明如下：

#### 数据准备
1. 从 `myIP.txt` 提取可用的 IP:Port 池：
   ```bash
   grep -oP 'vless://[^@]+@\K[0-9.]+:[0-9]+' "$TARGET_MYIP"
   ```

2. 从 `myDizhi.txt` 提取需要处理的节点：
   ```bash
   grep -E '^(vless|vmess|trojan)://' "$TARGET_MYDIZHI"
   ```

#### 节点处理与替换

**VMess节点处理**：
1. 解码 Base64 格式的配置信息
2. 检查是否包含 "gemini" 标记，包含则跳过
3. 从 IP 池中按顺序获取新 IP:Port
4. 更新 JSON 配置中的 `add`（IP）和 `port` 字段
5. 重新编码为 Base64 格式

**VLESS/Trojan节点处理**：
1. 使用正则表达式解析链接结构
2. 检查是否包含 "gemini" 标记，包含则跳过
3. 从 IP 池中按顺序获取新 IP:Port
4. 替换原链接中的 IP 和端口部分
5. 保持其他参数不变

#### IP池使用策略
- 采用循环轮询方式使用 IP:Port 池
- 每处理一个节点，计数器递增
- 通过 `计数器 % IP池大小` 实现循环分配

#### 特殊处理
- 跳过所有包含 "gemini" 关键字的节点
- 保留原始节点文件，创建修改后的新文件
- 分离节点内容和配置注释

### 3. 文件输出

系统会生成以下文件：

- `timestamp.txt`：记录各节点文件最后更新时间
- `myDizhi_modified.txt`：IP替换后的节点文件
- `dizhiR.txt`：运行报告，包含耗时和节点新鲜度
- `dizhi123.txt`：合并所有节点和信息的最终文件

### 4. 报告与监控

#### 时间戳管理
- 跟踪每个节点文件的变更时间
- 只在文件实际变更时更新时间戳
- 计算节点新鲜度（天数）

#### 报告生成
- 生成包含运行时间和耗时的报告链接
- 计算各类型节点的更新天数差
- 以特殊 trojan 链接形式展示报告信息

## 注意事项

1. **Secrets 配置**：
   - `VMESS_URL`：VMess 节点源地址
   - `SS_URL`：SS/SSR 节点源地址
   - `TARGET_REPO`：目标仓库地址
   - `TARGET_PAT`：访问目标仓库的 Personal Access Token
   - `TARGET_FOLDER`：目标仓库中的文件夹路径

2. **依赖项**：
   - 系统需要安装 `jq` 工具处理 JSON 数据

3. **文件结构**：
   - 目标仓库需包含 `myIP.txt` 和 `myDizhi.txt` 文件
   - 确保有足够的权限进行读写操作

## 维护说明

- 系统会自动清理旧的 workflow 运行记录
- 每次运行都会生成详细的调试信息
- 如果没有变化，不会创建空的提交

## 使用方法

1. 配置必要的 Secrets
2. 确保 GitHub Actions 权限设置正确
3. 手动触发一次工作流进行测试
4. 确认输出文件符合预期后，即可依靠定时任务自动运行
